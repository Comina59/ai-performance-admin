# 整体设计文档：AI+绩效考核系统

- 日期：2026-06-05
- 版本：v1.0
- 关联需求：docs/project/project20260202.md

## 1. 功能边界

### 1.1 首版范围

围绕固定业务链路实现完整绩效考核流程：

**员工发起当月绩效 → 领导审批评分项 → 员工自评 → 领导独立评分 → 差异判定 → 触发面谈 → 输出改进计划 → 周期结束**

### 1.2 首版包含

| 功能域 | 说明 |
|--------|------|
| 考核关系管理 | 独立维护员工-领导考核关系，支持多级领导完整层级链，含历史表 |
| 考核配置管理 | 绩效管理员配置差异阈值、考核周期类型等系统参数 |
| 绩效单全流程 | 发起、审批、自评、评分、差异判定、面谈、改进计划确认 |
| 评分项管理 | 员工手动填写评分项（主子表模式），权重校验 |
| 差异判定 | 可配置阈值（单项差异比例、总分差），自动触发面谈 |
| 改进计划 | 强制改进计划，所有流程结束后必须确认 |
| 历史查询 | 支持查看往月绩效记录 |
| 多考核周期 | 支持周/月/季/年四种考核周期 |

### 1.3 首版不含

| 功能 | 说明 |
|------|------|
| AI辅助功能 | 润色描述、检查完整性、生成评语、生成面谈提纲/总结等，后续迭代 |
| 消息通知 | 站内消息通知，后续迭代 |
| 附件上传 | 自评附件上传，后续迭代 |
| 薪酬联动 | 不做薪酬自动联动 |
| 跨周期对比 | 不做跨周期复杂对比分析 |
| 多人联合评分 | 仅员工+直属管理者 |
| 移动端App | 先Web/H5 |

## 2. 角色

| 角色 | 职责 | 说明 |
|------|------|------|
| 普通员工 | 发起绩效、填写评分项、自评、查看结果、参与面谈、确认改进计划 | 仅操作自己的绩效单 |
| 领导 | 审批评分项、独立评分、面谈、确认改进计划 | 仅操作直属下属的绩效单，多级领导中仅直属领导有操作权限，上级领导可查看 |
| 绩效管理员 | 查看进度统计、配置系统参数、异常处理（如代为指定领导） | 全局视角，不参与打分 |

## 3. 全局状态流转

```
草稿中(01) → 待领导审批(02) → 待员工自评(03) → 待领导评分(04) → 差异判定中(05)
    ↑              |                                              |
    |              ↓                                              ↓
    |         (驳回回草稿)                              待绩效面谈(06) / 待改进计划确认(07)
    |                                                      |
    |                                                      ↓
    └─────────────────────────────────────────── 待改进计划确认(07) → 已完成(08)
```

### 状态枚举

| 状态码 | 状态名 | 可执行操作（员工） | 可执行操作（领导） |
|--------|--------|-------------------|-------------------|
| 01 | 草稿中 | 编辑评分项、保存草稿、提交审批 | — |
| 02 | 待领导审批 | — | 审批通过、驳回 |
| 03 | 待员工自评 | 填写自评、保存草稿、提交自评 | — |
| 04 | 待领导评分 | — | 填写评分、保存草稿、提交评分 |
| 05 | 差异判定中 | — | —（系统自动判定） |
| 06 | 待绩效面谈 | 查看面谈安排 | 发起面谈、完成面谈 |
| 07 | 待改进计划确认 | 确认计划、申请调整 | — |
| 08 | 已完成 | 查看结果 | 查看结果 |

### 状态流转规则

| 当前状态 | 目标状态 | 触发条件 | 操作角色 |
|----------|----------|----------|----------|
| 01 | 02 | 员工提交审批（权重合计=100） | 员工 |
| 02 | 03 | 领导审批通过 | 领导 |
| 02 | 01 | 领导驳回 | 领导 |
| 03 | 04 | 员工提交自评（校验通过） | 员工 |
| 04 | 05 | 领导提交评分 | 领导 |
| 05 | 06 | 差异判定触发阈值 | 系统 |
| 05 | 07 | 差异判定未触发阈值 | 系统 |
| 06 | 07 | 领导完成面谈并生成改进计划 | 领导 |
| 07 | 08 | 员工确认改进计划 | 员工 |

## 4. 核心流程

### 4.1 员工发起绩效

1. 员工点击"新建本月绩效"，系统校验同一员工同一周期仅允许一个绩效单
2. 创建绩效单（状态=草稿中），进入评分项配置页面
3. 员工手动添加评分项（名称+权重），系统实时显示权重合计
4. 保存草稿或提交审批

### 4.2 领导审批评分项

1. 领导在团队工作台看到待审批项
2. 查看员工填写的评分项与权重
3. 审批通过 → 状态变为"待员工自评"
4. 驳回 → 填写驳回原因，状态回"草稿中"

### 4.3 员工自评

1. 员工在自评页面为每个评分项填写自评分、成果描述
2. 校验：自评分≤权重、≥0，成果描述非空
3. 提交自评 → 状态变为"待领导评分"

### 4.4 领导独立评分

1. 领导查看员工自评内容（左侧展示）
2. 领导独立填写每项分数和评语（右侧输入区，默认不带入员工分数）
3. 校验：每项分数≤权重
4. 提交评分 → 状态变为"差异判定中"，系统自动执行差异判定

### 4.5 差异判定

系统自动判定规则（阈值可配置）：
- 规则1：任一分项差异比例 ≥ 配置阈值（默认30%），差异比例 = |领导分-员工分| / 权重
- 规则2：总分差 ≥ 配置阈值（默认20分）
- 满足任一即触发面谈 → 状态变为"待绩效面谈"
- 均不满足 → 状态变为"待改进计划确认"

### 4.6 绩效面谈

1. 领导发起面谈，填写面谈纪要和总结
2. 生成改进计划（可编辑）
3. 完成面谈 → 状态变为"待改进计划确认"

### 4.7 改进计划确认

1. 员工查看改进计划（改进项、负责人、截止日期、验收标准）
2. 确认计划 → 状态变为"已完成"
3. 申请调整 → 填写原因提交给领导，领导重新编辑改进计划

## 5. 数据模型

### 5.1 核心实体

| 实体 | 说明 | 关键字段 |
|------|------|----------|
| perf_assess_relation | 考核关系配置表 | 员工ID、领导ID、层级序号、考核类型ID、生效周期 |
| perf_assess_relation_history | 考核关系历史表 | 同上+快照时间，保存已发生考核的隶属关系 |
| perf_assess_config | 考核配置表 | 配置项KEY、配置值、配置描述 |
| perf_assess_type | 考核类型表 | 类型名称（周/月/季/年）、状态 |
| perf_assess | 绩效单主表 | 员工ID、考核类型ID、周期、状态、截止日期 |
| perf_assess_item | 评分项子表 | 绩效单ID、评分项名称、权重、员工自评分、员工成果描述、领导评分、领导评语 |
| perf_assess_interview | 面谈记录表 | 绩效单ID、面谈纪要、面谈总结、面谈时间 |
| perf_assess_improve | 改进计划表 | 绩效单ID、改进项、负责人、截止日期、验收标准、确认状态 |
| perf_assess_reject | 驳回记录表 | 绩效单ID、驳回原因、驳回人 |

### 5.2 实体关系

```
perf_assess_type 1──N perf_assess_relation（考核类型下配置考核关系）
perf_assess 1──N perf_assess_item（绩效单包含多个评分项）
perf_assess 1──1 perf_assess_interview（绩效单关联一次面谈）
perf_assess 1──N perf_assess_improve（绩效单包含多个改进项）
perf_assess 1──N perf_assess_reject（绩效单可能多次驳回）
perf_assess_relation 1──N perf_assess_relation_history（考核关系变更产生历史记录）
```

## 9. 替代方案与决策记录

| 决策点 | 方案A | 方案B | 选择 | 理由 |
|--------|-------|-------|------|------|
| 模块拆分 | 单模块 | 双模块拆分 | 方案A | 功能强内聚，拆分增加复杂度无收益 |
| 考核关系 | 单层+递归 | 完整层级链 | 方案B | 需求明确要求完整层级链，避免递归查询性能问题 |
| 评分项关系 | 主子表 | 独立实体 | 方案A | 评分项随绩效单生命周期，无复用需求 |
| 流程驱动 | 状态机驱动 | 分散操作 | 方案A | 绩效流程强状态依赖，集中管控更安全 |
