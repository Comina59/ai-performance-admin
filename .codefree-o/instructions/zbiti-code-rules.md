# 最高准则：核心开发方法论（Superpowers 工作流深度融合）

## 首要原则

**收到任何开发任务时，必须严格按照下文阶段流转，启用每阶段的【入口自检】与【出口自检】机制。系统化方法论永远优于凭直觉直接编码。门控阶段未通过时，任何实施动作均禁止。**

> 技能优先级：流程技能（构思设计、调试定位） > 实施技能（TDD、代码审查）。

> 主Agent定位：**编排者，非实施者**。主Agent只负责调度、状态同步、自审、决策；业务代码的批量探索、编写、修改一律交由子Agent或 skill 工具。
>
> **主Agent 可直接 Read 的文件白名单**：
> - Plan 文档（`docs/plans/*-plan.md`）
> - 执行日志（`docs/plans/*-execution-log.md`）
> - 设计文档（`docs/plans/*-design.md`、`*-API-*-design.md`）
> - 配置文件（`*.yml`、`*.xml`（pom）、`.opencode/**`、`AGENTS.md`）
> - 用户在对话中直接粘贴 / 上传的代码片段
>
> **主Agent 可有限读取业务源代码的例外场景**（用于调试与验证，不得超出范围）：
> 1. 调试 mvn / 测试失败时，根据 verifier 输出的栈顶 ≤ 3 个文件，定位错误行（用 `read` 工具且指定 offset/limit，禁止全文件读）
> 2. 用户在对话中明确点名某个文件让主Agent 评估
> 3. 阶段4 汇报时执行 `git diff --stat`（只看变更摘要，不全文 diff）
>
> ⛔ 仍然禁止：批量扫描业务源码、用 `read` 一次读完整 Service/Controller、为了"了解一下"而读源码。批量探索必须走 `zbiti-code-reader`。代码读取路径限定在 `src/main/` 目录下，禁止读取构建产物或依赖缓存目录（详见「全局约束」）。

## 总则

### 文档与日志路径规范

设计文档、计划文档、执行日志统一保存到 `docs/plans/` 目录：

| 文档类型 | 命名规则 | 用途 |
|---------|---------|------|
| 整体设计文档 | `YYYYMMDD-中文功能名-design.md` | 阶段1产出，描述功能整体设计 |
| API 子设计文档 | `YYYYMMDD-中文功能名-API-{API中文名}-design.md` | 阶段1产出，当存在 ≥1 个 API 时强制，⛔ 禁止将 API 设计内联到整体设计文档 |
| 实施计划文档 | `YYYYMMDD-中文功能名-plan.md` | 阶段2产出，包含任务分解表 |
| **执行日志文档** | `YYYYMMDD-中文功能名-execution-log.md` | **阶段3产出，每完成一个 TODO 立即追加一条记录** |
| 前端 API 文档 | `YYYYMMDD-中文功能名-frontend-api.md` | 阶段4产出 |
| 前端开发指引 | `YYYYMMDD-中文功能名-frontend-guide.md` | 阶段4产出 |

> 功能名必须使用中文，不得使用英文或拼音。所有文档创建/更新后必须立即 `git add`。

### 硬门控协议

- 阶段0、阶段1、阶段2均为硬门控阶段，仅产出文档，不产生代码。可按需通过 `skill` 工具加载知识查询型 Skill（zbiti-* 系列，注意一定不是 *-agent 结尾），但⛔禁止调用任何实施型子Agent。
- 用户明确回复"批准/确认/继续"后方可推进至下一阶段。
- 若任务原始复杂度为"简单"，阶段1和阶段2可合并为一次快速设计（仍需输出最小化的设计要点和任务列表，无需生成独立整体设计文档，⛔ 但若存在 API 仍必须独立生成 API 子设计文档），**但必须生成极简 Plan 文档（仅含：目标、原始复杂度、任务分解表、验收标准）和空 Log 骨架**，用户确认后可进入阶段3。

### 错误熔断

子Agent 报错的处理路径分为两层：

1. **可诊断的失败**（编译错误、测试失败、明确的代码缺陷）→ 走 3.9 失败处理（自动重试 1 次 + 调试修复 1 次，总调用上限 3 次）
2. **不可诊断的失败**（子Agent 异常退出、返回乱码、超时、环境/网络错误等）→ 触发本熔断：立即终止当前任务链 → 向用户报告错误信息及上下文 → 等待用户决策（重试/跳过/回滚），⛔ 不自动恢复、⛔ 不自动重试超过 1 次

### 阶段自审通用模板

每个阶段进入前与退出前，主Agent必须显式输出自审清单。**结论为 PASS 时压缩为单行**，仅当 BLOCK/FAIL 时展开完整清单：

**PASS 格式**：
```
[阶段X 入口/出口自检] PASS
```

**BLOCK/FAIL 格式**（展开完整清单）：
```
[阶段X 入口自检]
- [ ] 前置阶段产出物已就绪：{文档/状态}
- [ ] 前置阶段已获用户批准：{是/否}
- [ ] 当前阶段所需上下文已加载：{Plan片段 / 设计文档 / 执行日志}
结论：BLOCK（原因：...）
```

```
[阶段X 出口自检]
- [ ] 本阶段必产文档已生成且 git add：{文件列表}
- [ ] 本阶段约束规则已全部遵守：{逐条对照}
- [ ] Plan 文档「阶段状态」已更新：{阶段X → ✅}
- [ ] 执行日志已追加阶段总结条目：{是/否/N/A}
结论：FAIL（原因：...，修复动作：...）
```

⛔ 自审输出 FAIL 时禁止推进，必须先修复。

## 阶段 0：需求范围审查（单功能原则）

### 入口自检

```
[阶段0 入口自检]
- [ ] 已获取用户原始需求完整描述
- [ ] 未直接进入设计或编码
结论：PASS
```

### 执行内容

评估用户原始需求的边界：

- **多模块/多无关功能**（如同时要求"用户管理"和"库存管理"；或"添加订单"与"生成报表"）：
  - ⛔ 立即阻断，不得进入阶段1
  - 明确回复：本 Agent 仅面向相对独立、高内聚的单个功能或模块进行设计与开发，请将需求拆解后分别提交
  - 提供拆解建议
- **单一模块内的功能扩展** 或 **单功能内多个紧密关联的子操作**（如单模块 CRUD、单业务流程"下单并支付"、相互强依赖的功能集合）：
  - 允许进入阶段1

**判定标准**：

- 多模块：涉及不同业务领域（用户/商品/订单）且模块间无直接上下游依赖
- 多个无关功能：功能间无共享数据模型或强业务约束
- 允许场景：单 CRUD、单业务流程、强依赖功能集合

### 出口自检

```
[阶段0 出口自检]
- [ ] 已明确判定属于"允许场景"还是"阻断场景"
- [ ] 若阻断，已向用户输出拆解建议并停止后续动作
- [ ] 若通过，已记录判定理由备阶段1引用
结论：PASS / FAIL
```

## 阶段 1：Brainstorming（构思设计）

### 入口自检

```
[阶段1 入口自检]
- [ ] 阶段0已通过且记录判定理由
- [ ] 已识别需求涉及的领域（用于决定加载哪些 zbiti-* skill）
- [ ] 当前会话上下文中无大段业务源代码（若有，先清理）
结论：PASS / BLOCK
```

### 执行内容

1. **提问澄清**：结构化问题帮助用户明确需求，⛔ 一次只问一个问题，禁止一次抛出大量问题，⛔ 禁止直接给代码。当存在多组要确认信息时，可使用 **tag 页签** 格式分组展示，用户可切换查看与回答（如：发票主表包含哪些字段、发票子表包含哪些字段）。
2. **领域知识查询**：通过 `skill` 工具按需加载相关知识 Skill（如 `zbiti-code-reader` 等）。⛔ 除非用户明确要求，否则不要读取本地业务源代码
3. **本地代码探索（仅当必要时）**：必须通过 `zbiti-code-reader` skill 三步分层调用（见下方"分层调用规范"），⛔ 主Agent禁止直接使用 Read 工具读取业务 Java 文件。⛔ 若在 `src/main/` 目录下未找到目标文件，**禁止回退读取 `target/` 目录或尝试反编译 jar 包**，应改为向用户确认文件位置或扩大搜索范围
4. **探索替代方案**：⛔ 强制要求，不可省略——至少提出 2-3 种实现路径，说明各自取舍，并在需求澄清过程中与用户确认所选方案。未提出替代方案前禁止进入步骤5
5. **输出整体设计文档**：保存到 `docs/plans/YYYYMMDD-中文功能名-design.md`
   - 内容：功能边界、核心流程、数据模型、API 列表（若有）、模块归属、依赖的 zbiti SDK
6. **多 API 二次澄清**（当整体设计中包含 ≥1 个 API 时强制触发）：
   - 逐个对每个 API 进行独立的需求澄清（一次一个问题）
   - **每个 API 必须独立生成子设计文档**，⛔ 禁止将 API 设计内联到整体设计文档中
   - 子设计文档命名：`YYYYMMDD-中文功能名-API-{API中文名}-design.md`
   - 子设计文档须包含：端点定义、请求/响应结构（字段明细表）、**业务规则约束表**、校验逻辑、错误码、依赖服务、与其他 API 的交互关系、响应示例
   - 在整体设计文档的 API 列表中添加指向各子设计文档的相对路径链接
   - **API 分级**：将 API 分为 A 类（标准 CRUD，遵循 ICommonController 模式）和 B 类（非标业务逻辑，含状态流转/计算/外部集成）。
     - **判定细则**：若 API 仅包含 ICommonController 标准操作（list/add/update/delete/get）且无额外业务分支，归为 A 类；若 API 在标准操作基础上增加了状态校验、权限判断、外部调用等任一非标逻辑，归为 B 类；存疑时默认归 B 类

> **设计约束**：
> - API 设计必须严格参考项目现有的 Controller 接口模式（如 `ICommonController` 通用REST控制器接口），保持风格、请求方式、出入参格式一致；（HTTP方法可选：GET/POST；除非用户明确指定，否则不提供PUT/DELTE）
> - 命名区分：`-API-{API中文名}-design.md` 是阶段1的后端设计规格，`-frontend-api.md` 是阶段4的前端参考文档，二者用途不同，禁止混淆
> - ⛔ 禁止将 API 设计内容内联到整体设计文档中，每个 API 必须独立生成子设计文档
> - ⛔ API 子设计文档的请求/响应结构必须以**字段明细表**形式输出，禁止仅列出 JSON 示例或字段名列表。字段明细表格式如下：

| 字段名 | 中文名 | 类型 | 长度 | 必填 | 默认值 | 取值范围 | 说明 |
|--------|--------|------|------|------|--------|----------|------|
| xxx | xxx名称 | String | 64 | 是 | 无 | — | 业务描述 |
| status | 状态 | Integer | — | 否 | 0 | 0=禁用, 1=启用 | 枚举类型须列出所有候选值 |

>   - **中文名**：每个字段必须提供中文描述，便于前后端理解业务含义
>   - **类型**：Java 类型（String / Integer / Long / BigDecimal / Date / List\<T\> 等）
>   - **长度**：字符串类型须标注最大长度；数值类型可标"—"；集合类型标注最大元素数（如有约束）
>   - **必填**：是/否
>   - **默认值**：无默认值填"无"；有默认值须明确写出
>   - **取值范围**：枚举类型⛔**必须列出所有候选值**（如 `0=禁用, 1=启用`）；数值范围标注上下界（如 `1-999`）；无约束填"—"
>   - **说明**：补充业务规则、校验逻辑、与其他字段的关联等
>   - 嵌套对象（如请求体中的子对象）须单独用一张字段明细表描述，并在外层字段的"说明"列中标注"详见下表"
> - ⛔ API 子设计文档**必须包含业务规则约束表**，格式如下：
>
>   | 规则编号 | 触发条件 | 执行逻辑 | 影响字段 | 说明 |
>   |----------|----------|----------|----------|------|
>   | R1 | applyStatus IN ('01','04') | 允许取消，设置 applyStatus='05' | applyStatus | 仅草稿和已驳回可取消 |
>   | R2 | 乐企接口返回成功 | 更新申请状态为06 | applyStatus | 必须在乐企成功后更新 |
>
>   - **触发条件**：必须明确到具体字段值（如 `applyStatus IN ('01','04')`），禁止模糊描述（如"特定状态"）
>   - **执行逻辑**：必须写明具体操作（如"设置 applyStatus='05'"），禁止仅写"更新状态"
>   - **影响字段**：列出被修改的字段名及目标值
>   - **说明**：补充业务背景、与其他规则的互斥/优先关系
>   - ⛔ 状态码枚举值必须在业务规则约束表中明确列出（如 `01=草稿, 02=审批中`），禁止用模糊描述
>   - ⛔ B 类 API 的业务规则约束表必须覆盖所有分支逻辑；A 类 API 若无额外业务规则可标注"无额外业务规则"
>
> - ⛔ API 子设计文档**必须提供响应示例**，包含成功响应和失败响应两种场景，格式如下：
>
>   **成功响应示例**：
>   ```json
>   {
>     "code": 200,
>     "msg": "操作成功",
>     "data": { ... }
>   }
>   ```
>   说明：[描述成功时 data 中返回的关键字段及业务含义]
>
>   **失败响应示例**：
>   ```json
>   {
>     "code": 500,
>     "msg": "具体错误信息",
>     "data": null
>   }
>   ```
>   说明：[描述触发失败的场景条件、错误码含义、前端应对方式]
>
>   - 响应示例中的 JSON 结构须与字段明细表中定义的响应结构严格一致
>   - 失败响应示例须覆盖该 API 的主要业务异常场景（如：参数校验失败、业务规则冲突、资源不存在等），每个场景独立提供示例与说明
>   - `code` 和 `msg` 的取值须与 `ICommonController` 统一响应格式保持一致
>   - ⛔ 禁止仅提供一个笼统的失败示例（如只写"参数错误"），必须按业务异常场景逐一列举

### 本地项目代码读取必须使用 zbiti-code-reader Skill 进行分层调用

调用 `zbiti-code-reader` 时，**必须遵循以下三步流程**，禁止一步到位读取大量文件：

1. **结构探索**：首次调用必须使用 `summary_mode: "tree"`，目标为顶层目录（如 `src/main/java/com/xxx/module`），获取文件树
2. **筛选核心文件**：从 `tree` 输出中选择 **不超过3个** 核心文件（如 `*Service.java`、`*Controller.java`）
3. **深度读取**：对选中的每个文件，单独调用 `summary_mode: "signature_only"`（或 `full`），**一次只传一个文件路径**

> ⛔ 禁止对目录直接使用 `signature_only` 或 `full` 模式
> ⛔ 禁止一次性传入超过3个文件路径到单次调用（除非 `summary_mode` 为 `tree`）
> ⛔ 若 `zbiti-code-reader` 输出被截断，必须重新评估筛选范围而非简单增加文件数
> ⛔ 主Agent 禁止用通用 Read 工具读取业务源代码，必须经由 `zbiti-code-reader`
> ⛔ **路径范围与回退禁止**：传入路径必须限定在 `src/main/` 下，若 `zbiti-code-reader` 返回的文件路径包含构建产物或缓存目录，应跳过并仅关注 `src/main/` 下的源码（完整目录黑名单见「全局约束」）。⛔ 若在 `src/main/` 下未找到目标文件，**禁止回退到 `target/`、`build/` 等构建产物目录查找，禁止尝试从 jar 包反编译获取源码**。正确做法：①向用户确认文件是否存在及正确路径；②扩大搜索范围但仍在 `src/main/` 内；③通过 `skill` 工具查询平台 SDK 知识库获取所需类信息

### 出口自检

```
[阶段1 出口自检]
- [ ] 整体设计文档已生成并 git add
- [ ] 存在 API 时，所有 API 子设计文档已独立生成并 git add（⛔ 禁止内联到整体设计文档）
- [ ] API 子设计文档中已包含业务规则约束表（B 类 API 须覆盖所有分支逻辑，A 类 API 可标注"无额外业务规则"）
- [ ] API 已分级（A 类/B 类）
- [ ] 设计文档中标注了所有依赖的 zbiti SDK 及模块归属
- [ ] 整体设计文档的 API 列表已链接到各子设计文档
- [ ] 已提出 ≥2 种替代方案并与用户确认所选方案（⛔ 不可省略）
- [ ] 已明确请求用户批准
- [ ] 用户已回复"批准/确认/继续"（未回复前禁止进入阶段2）
结论：PASS / FAIL
```

### 产出物

- 单个整体设计文档（必选）
- 若干 API 子设计文档（当 API 数量 ≥1 时必选，⛔ 每个 API 独立一份，禁止内联）

整体设计文档获批后自动激活阶段2；若存在子设计文档，需所有子设计文档均获批后方可进入阶段2。

## 阶段 2：Writing Plans（制定实施计划）

### 入口自检

```
[阶段2 入口自检]
- [ ] 阶段1所有产出文档均已获用户批准
- [ ] 已读取所有阶段1产出文档作为输入
- [ ] 已明确「原始复杂度」初步判断
结论：PASS / BLOCK
```

### 执行内容

设计确认后（含所有多 API 子文档），**必须输出结构化的实施计划文档**。

保存到 `docs/plans/YYYYMMDD-中文功能名-plan.md`，内容必须包含下列所有章节，**任何"待补充"字样均视为 FAIL**：

```markdown
# 实施计划：[功能名称]

## 目标
[一句话描述]

## 原始复杂度（一旦确定，后续不可变更）
- [ ] 简单（单文件改动 或 仅配置/文档/SQL 调整，不涉及业务逻辑分支）
- [ ] 中等（2-4 个文件，涉及一个 service 方法及其调用链；或单表 CRUD 全套）

### 3.0 执行日志文件规范（核心追踪机制）

> **目的**：将主Agent状态完全外化到文件，让上下文压缩后可零损失恢复，让用户可随时查看进度。

**执行日志文件位置**：`docs/plans/YYYYMMDD-中文功能名-execution-log.md`

**文件结构**（在阶段2创建空骨架时即固定）：

```markdown

**强制写入时机**（每条规则单独触发一次文件追加操作）：

1. ⏳ → ✅：子Agent完成 + 两层审查通过 + 满足验收标准后，**立即追加** TODO 记录段
2. ⏳ → ❌：失败后立即记录失败原因 + 最后一次错误输出摘要（≤20行）
3. 阶段切换：每次进入新阶段，覆盖更新「阶段进度快照」

- ⛔ 禁止批量补登（如一次性写10个已完成任务）
- ⛔ 禁止仅更新 Plan 不更新 Log，二者必须同步
- ⛔ 禁止省略「变更文件」「验收结果」「审查结论」三个关键字段
- ✅ 每次写入后必须 `git add docs/plans/`

### 3.1 压缩恢复协议（Compression Recovery Protocol）

> **触发条件**：主Agent检测到上下文已被压缩（历史消息截断、summary 替换完整对话）、收到形如"继续"的简短指令但无前文上下文、或发现 Plan/Log 中的状态与当前记忆不一致。

> **注**：所有任务（含简单复杂度）均强制生成 Plan + Log（详见总则「硬门控协议」），本协议恒适用。若极少数场景下 Plan/Log 缺失（如主Agent 临时直接处理 ≤ 5 行配置改动），仍需先按下方步骤 2 列目录确认。

**强制动作序列**（顺序不可调换）：

1. **暂停所有新调度**：未恢复完成前禁止启动任何子Agent
2. **识别功能名**：如果当前会话无法确定功能名，必须执行 `ls docs/plans/ -t | head -10`（或 PowerShell 等价命令）列出最近的 `*-plan.md` 与 `*-execution-log.md`，并向用户确认"是否继续 {最新功能名}"。⛔ 禁止凭猜测继续
3. **读取 Plan 文档**：`Read docs/plans/YYYYMMDD-中文功能名-plan.md`，获取任务分解表 + 阶段状态 + 原始复杂度
4. **读取执行日志**：`Read docs/plans/YYYYMMDD-中文功能名-execution-log.md`，获取「阶段进度快照」+ 最近 N 条 TODO 记录
5. **状态对账**：以"Plan 状态 + Log 记录"的交集为唯一真实来源，清空内存中所有"我记得做过什么"的推断
6. **构造待执行队列**：所有 Plan 中状态 ≠ ✅ 且 Log 中无 ✅ 记录的任务进入队列
7. **校验阶段位置**：
   - 仍有未完成任务 → 继续阶段3
   - 全部 ✅ → 进入阶段4
8. **输出恢复确认**：

```
[压缩恢复完成]
- 功能：{中文功能名}
- 原始复杂度：{简单/中等/复杂}
- 当前阶段：{阶段3}
- 已完成任务：T1, T2（共 2 个）
- 待执行任务：T3, T4（共 2 个）
- 下一步动作：调度 {子Agent} 执行 T3
```

⛔ **禁止行为**：
- 凭记忆推断"剩下哪些任务"
- 只读 Plan 不读 Log（Plan 可能领先于实际成果）
- 只读 Log 不读 Plan（Log 可能缺失阶段总状态）
- 跳过「输出恢复确认」直接调度

### 3.2 执行调度规则（融合模式）

**三层职责**：

| 层次 | 角色 | 说明 |
|------|------|------|
| **调度层** | 主Agent | 任务排序、并行策略、状态同步、日志写入、变更暂存 |
| **执行层** | 专用子Agent（按路由表选择），路由表未覆盖时允许 `general` 兜底 | 路由表覆盖的操作⛔禁止用 `general` 替代 |
| **审查层** | `general` 类型子Agent | 两层审查（spec + quality），不使用专用Agent |

**标准执行流程（每个 TODO 必须完整经过）**：

```
1. 主Agent：调度前自检（路由匹配 + 依赖检查 + 输入完整性） → 输出 [TODO 入口自检]
2. 主Agent：声明路由匹配并调度专用子Agent
3. 专用子Agent：执行任务并返回结果（文件路径、变更摘要）
4. 主Agent：调度 general 子Agent 执行【第一层 spec 审查】
5. 主Agent：第一层 PASS 后调度 general 子Agent 执行【第二层 quality 审查】
6. 主Agent：两项全 PASS 后，原子操作 —— 
   a. 更新 Plan 文档对应任务状态 ⏳→✅
   b. 追加 execution-log.md 一条 TODO 记录段
   c. git add docs/plans/
7. 主Agent：输出 [TODO 出口自检]，进入下一任务
```

> **验收策略说明**：每个编码任务的验收标准是"是否满足设计要求"而非"编译通过"。编译验证统一由阶段3末尾的集成验证任务（`zbiti-verifier-agent`）完成，不在此流程中逐任务执行编译。详见 3.8 节。

⛔ 任一环节失败：立即停止后续步骤，将状态写为 ❌ 并在 Log 中记录失败摘要，触发 3.9 失败处理。

### 3.3 TODO 入口/出口自检（每个任务必做）

**TODO 入口自检模板**：

结论为 PASS 时压缩为单行：`[TODO {ID} 入口自检] PASS`；仅当 BLOCK 时展开完整清单：

```
[TODO {ID} 入口自检]
- [ ] 路由匹配已完成：{匹配结果}
- [ ] 前置依赖已全部 ✅：{依赖任务ID列表}
- [ ] 输入上下文完整：{Plan行号 / 设计文档 / 涉及文件}
结论：BLOCK（原因：...）
```

**TODO 出口自检模板**：

结论为 PASS 时压缩为单行：`[TODO {ID} 出口自检] PASS`；仅当 FAIL 时展开完整清单：

```
[TODO {ID} 出口自检]
- [ ] 子Agent已返回执行结果
- [ ] 两层审查已通过（或微变更自审已通过）
- [ ] Plan 任务状态已更新 ⏳→✅
- [ ] execution-log.md 已追加 TODO 记录段
- [ ] git add docs/plans/ 已执行
结论：FAIL（原因：...，修复动作：...）
```

### 3.4 系统性调试纪律

⛔ 遇到测试失败/运行时错误时，**禁止凭直觉直接修改代码**，必须：

1. **不要猜测**：先稳定复现，读取完整错误输出
2. **逆向追踪**：从错误触发点向上追溯调用栈定位根因
3. **修复根因**：仅修改根本原因，不补丁式修复表面症状
4. **验证修复**：重新运行确认问题不再出现且无新问题引入

> 调试证据（错误摘要 + 根因结论）必须写入 execution-log.md 对应任务的"关键决策/异常"字段。

### 3.5 并行执行规范

> **RuoYi 项目实情**：建表 → 编写业务代码（Controller/Service/Mapper/Domain/XML）是强串行链路，真正可并行的场景多见于"多个独立 service 方法/多个独立 mapper xml"。⛔ 跨依赖任务严禁并行。

**判定并行性**：仅当任务在 Plan 中标记 `可并行=✅` 且依赖列为「无」时才可同时调度。

| 同时可调度的子Agent 数 | 启动方式 |
| :---------- | :------------ |
| 1 个        | 串行 |
| 2-3 个      | 全部并行 |
| 4+ 个       | 最多 3 个并行，剩余分批串行 |

**子Agent指令模板（主Agent构造时必填）**：

```markdown
## 子Agent任务
- **任务ID**：[T1]
- **关联 Plan**：docs/plans/YYYYMMDD-中文功能名-plan.md 第 T1 行
- **描述**：[具体任务，包含完整上下文]
- **涉及文件**：[精确路径列表]
- **依赖**：[已完成的前置任务 ID]
- **验收标准**：[按3.8分类填写，如"生成代码已迁移至业务模块"/"实现与设计文档一致"/"集成编译验证通过"]
- **必须遵守的全局编码规范**：见 zbiti-code-rules.md「全局编码规范」
- **禁止行为**：[根据任务上下文具体列出]
```

⛔ 子Agent指令中若缺少 「涉及文件」「验收标准」「禁止行为」任意一项 → 主Agent自身违规。

### 3.6 两层审查（每个 TODO 必做，⛔ 不可豁免）

每个专用子Agent执行完成后，主Agent必须按顺序调度 `general` 类型子Agent执行两层审查：

**第一层：规范一致性审查（Spec Compliance）**
- 检查实现是否与 Plan 和设计文档完全一致，无遗漏、无多余实现
- 存在多 API 子设计文档时，逐一核对每个 API 实现是否与其子文档一致
- 结论：`PASS` / `FAIL`

**第二层：代码质量审查（Code Quality）**
- 前置：第一层 PASS 后方可执行
- 检查项：逻辑正确性、命名规范、测试覆盖、边界处理、是否违反「全局编码规范」
- 结论：`PASS` / `WARNING` / `FAIL`

**问题处理**：
- 🔴 **Critical（FAIL）**：阻塞进度，调度 `zbiti-code-fix-agent` 修复后从第一层重新审查
- 🟡 **Warning**：记录到 execution-log.md「关键决策/异常」字段，可继续推进

**针对"简单"任务的特殊处理**：
- 仍必须调度 `general` Agent 执行两层审查，可合并为一次调用（同时检查规范与质量）
- 审查结论必须明确包含 `PASS` / `WARNING` / `FAIL`
- ⛔ 简单任务不得跳过审查直接标记完成

**针对"微变更"任务的快速通道**：
- 适用条件：变更 ≤ 5 行且不涉及业务逻辑（如单行 pom 依赖、yml 配置项、import 补充）
- 主Agent可直接执行并自审，无需调度子Agent审查，但必须在 execution-log 中记录变更内容与自审结论
- ⛔ 微变更任务仍须在 Plan 中有独立 TODO 项，不得省略

> **设计目的**：防止压缩后误判"简单任务"而跳过审查，导致低质量代码合并。

### 3.7 TDD 纪律

> **适用范围**：本项目采用标准 CRUD（generator 产出）⛔ 禁止补单测；TDD 仅适用于**纯手写的核心业务方法（含分支/计算/状态机/对外集成）**。

**触发条件（任一满足才执行 TDD）**：

1. 任务涉及 `if/switch/while` 等业务分支，且分支 ≥ 2
2. 任务包含金额、库存、状态流转等强约束计算
3. 任务调用外部系统（HTTP/MQ/第三方 SDK），需要桩测验证
4. 用户在需求中明确要求"必须有单测"

**TDD 流程**（命中触发条件时）：

1. **RED**：先写测试，验证失败（测试由 `zbiti-business-code-agent` 编写）
2. **GREEN**：写最少代码让测试通过
3. **REFACTOR**：重构优化，保持测试通过

execution-log.md 中需包含 RED→GREEN 的转换证明（测试文件路径 + 首次失败摘要 + 实现后通过摘要）。

**豁免清单**（**不要**为以下任务硬塞测试）：

- generator-code 生成的 Mapper/Domain/Controller/Service 基础骨架
- 纯 mybatis xml SQL 修改
- 配置文件、yml、pom 改动
- 文档、注释、命名调整
- 数据库 DDL / 序列 / 索引
- Configuration 类、SpringBoot autoconfigure 装配类

> 上述豁免任务的编译验证统一由阶段3末尾的集成编译验证任务完成，无需单独执行编译。

### 3.8 完成前强制验证

⛔ 任何 TODO 在标记 ✅ 之前必须通过对应类型的验收：

**验收标准按任务类型分类**：

| 任务类型 | 验收标准 | 说明 |
|----------|----------|------|
| 建表 / 执行 SQL | 表/序列创建成功，DDL 执行无报错 | 检查数据库对象是否存在 |
| 创建 / 修改 Maven 模块 | 模块目录结构符合规范，pom.xml 依赖正确 | 检查文件是否按要求创建，依赖是否完整 |
| 编写业务代码 | 实现与设计文档一致，逻辑正确 | spec+quality 两层审查通过 |
| **集成编译验证** | 编译验证通过 | ⛔ 仅此任务执行编译，见下方说明 |
| 代码审查与修复 | 修复后 spec+quality 审查重新通过 | 检查修复是否完整 |

**集成编译验证任务（必须作为阶段3的最后一个编码任务）**：

阶段3所有编码任务完成后，必须设置一个集成编译验证任务（调度 `zbiti-verifier-agent`）。若验证失败，由 `zbiti-code-fix-agent` 集中修复，修复后重新验证，直到全部通过。

**每个 TODO 完成后的同步操作**：

1. **同步 Plan 状态**：更新 Plan 文档中该任务状态 ⏳→✅；若该 TODO 完成后某阶段全部任务都已 ✅，则同步更新「阶段状态」
2. **追加 execution-log.md**：按 3.0 节模板补全完整记录
3. **校验一致性**：重新 `Read` Plan 与 Log 对应位置，确认状态已成功落盘
4. **git add docs/plans/**：暂存文档变更

> 反模式：仅凭"代码看起来没问题"标记完成；未实际审查就声称"应该没问题"；只更新 Plan 不写 Log。

### 3.9 失败处理

> **重试预算（硬上限，跨章节统一口径）**：每个 TODO 的累计子Agent 调用次数 ≤ 3（首次执行 + 失败自动重试 1 次 + 调试后修复执行 1 次）。达到上限仍失败必须停手向用户报告。

- 子Agent 失败 → 自动重试 1 次（与首次同输入；重试仍失败禁止第二次自动重试）
- 重试仍失败 → 暂停当前 TODO，执行 3.4 系统化调试流程定位根因，调度 `zbiti-code-fix-agent` 做一次定向修复（计入第 3 次调用）
- 第 3 次调用仍失败 → 在 execution-log.md 中追加完整失败报告（错误摘要 + 已尝试动作 + 根因推测），向用户报告并等待决策，⛔ 禁止继续自动尝试

## 子Agent路由协议

> **核心原则**：子Agent必须按阶段需要调度，禁止跨阶段提前调用。知识查询用 `skill` 工具，任务执行用 `task` 工具调度 `subagent_type`，⛔ 禁止混用。

### 路由表

| 触发场景 | 调度方式 | 说明 |
|----------|----------|------|
| ⛔ 由 commands 目录下的命令文件触发的操作 | **按命令文件声明的执行方式执行**；若命令文件未声明则按本表匹配 | 命令工作流优先级最高，⛔ 禁止用其他行覆盖 |
| 阶段1-2：知识查询 | 主Agent通过 `skill` 工具加载 | 获取领域知识，⛔ 禁止调用实施型子Agent |
| 阶段1-2：设计/Plan/Log 文档撰写 | 主Agent 直接 `write`/`edit` | 文档撰写是编排职责，⛔ 不调度业务子Agent 写文档 |
| 需要建表 / 执行 SQL | 子agent:`zbiti-database-operate-agent` | 所有数据库操作 |
| 创建 / 修改 Maven 模块 | 子agent:`zbiti-modules-create-agent` | 模块结构变更 |
| 编写业务代码 | 子agent:`zbiti-business-code-agent` | 具体业务逻辑实现（Controller/Service/Mapper/Domain/XML） |
| 代码审查与修复 | 子agent:`zbiti-code-fix-agent` | 配合 3.6 两层审查后的修复执行 |
| 编译 / 测试验证 | 子agent:`zbiti-verifier-agent` | 编译与测试验证 |
| 读取本地代码摘要 | skill:`zbiti-code-reader`（按 1.x 节三步分层） | ⛔ 主Agent 禁止批量 Read 业务源码 |
| 调试 mvn 报错时定位单文件 | 主Agent 用 `read` 工具读取栈顶文件指定行（offset/limit） | 例外允许，详见首要原则白名单 |
| 生成前端 API 文档 | 子agent:`zbiti-doc-generator-agent` | 文档命名：`docs/plans/YYYYMMDD-中文功能名-frontend-api.md` 及 `-frontend-guide.md` |
| 代码编写 / 修改后 | 主Agent直接执行 `git add` | 主Agent用 bash 工具直接操作，⛔ 不允许 commit |

### 门控规则

- 当前操作若由 commands 目录下的命令文件触发，且命令文件声明了执行方式，则**按命令文件声明执行**，⛔ 命令文件声明优先级高于本路由表
- 匹配路由表 → **必须**调度对应专用子Agent，⛔ 禁止主Agent直接执行（**豁免**：微变更任务，见下方微变更豁免条款）
- **微变更豁免条款**：当任务同时满足以下全部条件时，即使匹配路由表，主Agent也可直接执行并自审，无需调度子Agent：
  1. 变更 ≤ 5 行（计数口径：新增+修改的净增行数，不含空行和纯注释行）
  2. 不涉及业务逻辑（即：不含 if/switch/循环/计算/状态变更/外部调用，仅限 pom 依赖、yml 配置项、import 补充等纯声明式改动）
  3. 单个 TODO 项内不可拆分为多个微变更以规避子Agent调度（⛔ 反滥用：同一 Plan 中连续 ≥3 个微变更 TODO 应合并为一个任务走子Agent）
  - 豁免后仍须：在 execution-log 中记录变更内容与自审结论；在 Plan 中有独立 TODO 项；输出路由声明时标注 `微变更豁免`
- ⛔ 路由表覆盖的操作类型（业务代码、文档生成、数据库操作、模块创建、编译验证），**无论复杂度，必须走子Agent**
- 不匹配路由表 + 简单操作（单文件 ≤ 5 行纯配置调整）→ 主Agent直接执行，并显式声明 `路由无匹配(简单): 主Agent直接执行`
- 不匹配路由表 + 复杂操作（多文件 / 逻辑复杂）→ 主Agent调度 `general` 类型子Agent执行，并显式声明 `路由无匹配(复杂): 兜底调度 general`

### 执行前强制声明

任何产出性操作前，主Agent必须显式输出：

```
[路由声明]
- 场景：{具体场景}
- 匹配结果：路由匹配:{子Agent名} / 路由无匹配(简单) / 路由无匹配(复杂)
- 调度方式：{专用子Agent / general 兜底 / 主Agent直接执行}
```

⛔ 未声明就执行 = 违规，立即停止并补声明。

## 全局约束

- **Git add 必须触发**：任何文件创建 / 修改后，主Agent必须立即执行 `git add`（包括子Agent完成、模块创建、plans 文档创建 / 更新等场景）
- **Plan 与 Log 状态必须同步**：每个 TODO 完成后，必须同时更新 Plan 任务状态与追加 execution-log.md 记录，⛔ 二者不可只更新其一
- **子Agent 间禁止共享可变状态**：所有上下文通过 Plan / Log / 设计文档传递，禁止依赖会话内存
- **⛔ 禁止依赖未完成的任务就启动后续子Agent**：依赖项必须在 Plan 中状态为 ✅ 才可调度后置任务
- **⛔ 主Agent禁止批量读取业务源代码**：批量探索必须经由 `zbiti-code-reader` skill 三步分层流程；仅在「首要原则」白名单例外场景下允许有限读取
- **⛔ 代码读取路径限定在 `src/main/` 目录**：所有代码探索和读取操作（含 `zbiti-code-reader` 调用和调试时的 `read` 工具）均禁止读取 `target/`、`build/`、`.gradle/`、`node_modules/`、`generate/` 等构建产物或缓存目录；⛔ **严禁路径回退**：当在 `src/main/` 下未找到目标文件时，禁止回退到 `target/` 目录查找，禁止反编译 jar 包获取源码
- **⛔ 模块创建是业务代码的前置依赖**：若业务代码需要新的 Maven 模块，模块创建任务必须先完成并在 Plan 中标记 ✅，之后才可调度 `zbiti-business-code-agent`
- **⛔ 编译验证统一在集成验证任务中执行**：单个编码任务的验收标准是"是否满足设计要求"而非"编译通过"；阶段3末尾必须设置集成编译验证任务（`zbiti-verifier-agent`）；验证失败时由 `zbiti-code-fix-agent` 集中修复
- **微变更任务快速通道**：定义及适用条件见3.6「微变更快速通道」及门控规则「微变更豁免条款」

  - 任何状态变更立刻落盘到 Plan / Log，让文件成为唯一真实来源
